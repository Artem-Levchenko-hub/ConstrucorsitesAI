### vsegpt DeepSeek оркестратор — задеплоено на прод (4926a79)
Директива владельца: оркестратор = Opus (роль `director`), ВСЕ воркер-роли = DeepSeek через vsegpt.

Root cause почему deepseek не работал на live-сервере (а «прямые вызовы работали»): LiteLLM-путь к vsegpt.ru флапал — (1) разовый транзиентный TLS-handshake ConnectTimeout из прод-контейнера; (2) риск утечки трафика в UK-прокси (GEMINI_HTTPS_PROXY для Gemini), т.к. vsegpt.ru не был в NO_PROXY. proxyapi-deepseek вообще 404. При этом сама контейнерная сеть до vsegpt OK (DNS/TCP/TLS ~0.1s).

Fix: новый `providers/vsegpt.py` — прямой sync httpx в треде (`trust_env=False` + no-proxy `mounts`, по образцу `sber.py`), ретрай транзиентного handshake, strip `<think>`, `max_tokens=16384` (thinking-модель иначе обрезает свой ответ). Dispatch в `litellm_router.acompletion` + `streaming` pseudo-stream (новый `default_max_tokens`) ДО LiteLLM; мёртвые deepseek-v4 LiteLLM-записи убраны. `ROLE_MODEL_MAP`: director=Opus, воркеры=deepseek-v4-flash-thinking; audit=Sonnet + audit_retry=Opus оставлены (dormant, vision-судья + эскалация). Fallback `deepseek→Haiku→Sonnet`. compose прокидывает `ROLE_MODELS`/`FORCE_MODEL` (ops-рычаг без деплоя).

# 04. SEO + Domains + Payments — стратегический план

> Цель: превратить omnia-mvp из голого билдера в **универсальное окно** — пользователь
> заходит, описывает идею, и через 15 минут у него: сайт + домен + SSL +
> приём платежей. Один аккаунт, одна форма паспортных данных, одна цена в рублях.

Документ — единый источник правды для трёх связанных тем. Когда идём в код,
сначала правим этот файл, потом пишем тикеты.

## 2. SEO — план

### 2.1. Два разных SEO

У нас **две поверхности**, которые надо ранжировать раздельно:

1. **Маркетинговая страница** `https://constructor.lead-generator.ru/` — наш собственный сайт-продавец.
2. **Сгенерированные сайты пользователей** `https://<их-домен>` или `https://<slug>.omnia.ai` — продукт.

### 2.2. Phase A — маркетинговый сайт (делаем сейчас, ~1 день)

**Технически** (всё в `apps/web/`, уже частично сделано в этом коммите):

- [x] `app/layout.tsx` — `<title>`, `<meta description>`, OG, Twitter card, viewport, theme-color, canonical, alternates, applicationName, keywords (общие).
- [x] `app/sitemap.ts` — Next.js native, отдаёт `/sitemap.xml`. Сейчас 3 URL, будет расширен по мере добавления страниц.
- [x] `app/robots.ts` — отдаёт `/robots.txt`, скрывает `/projects`, `/api/`, `/_next/`.
- [x] `app/icon.svg` + `app/opengraph-image.tsx` — favicon (закрывает 404 в консоли) + динамическая OG-картинка через `next/og` (1200×630, без бинарника в репо).
- [x] JSON-LD: `Organization` + `SoftwareApplication` в root layout, `FAQPage` в Faq-компоненте → eligible для rich snippets в SERP.
- [x] `lang="ru"` на `<html>`.
- [x] `next/font` уже с `display: "swap"` — нет FOIT.

**Что ещё надо в Phase A** (до публичного запуска):

- [ ] Реальная OG-картинка как `app/opengraph-image.tsx` сейчас — generic. После брендинга можно заменить на дизайнерский PNG.
- [ ] Регистрация в **Yandex Webmaster** (https://webmaster.yandex.ru) → подтверждение через `<meta name="yandex-verification">` (env `NEXT_PUBLIC_YANDEX_VERIFICATION` уже встроен).
- [ ] Регистрация в **Google Search Console**.

- [ ] **Hreflang** добавить, если будем делать английскую версию (пока нет).
- [ ] Web Vitals — LCP/CLS/INP мониторить через `next/web-vitals` → отправлять в Метрику.

**Контент** (нужен копирайт + дизайн):

- [ ] Hero с одним сильным глаголом действия + видео-демо (3–5 сек preview генерации).
- [ ] Секция «Как это работает» — 3 шага: промпт → AI генерит → твой сайт онлайн.
- [ ] Секция кейсов — 3–5 примеров «вот сайт сделал за 12 минут» (после первых пользователей).
- [ ] Сравнение с Tilda/Wix/Bitrix — таблица «AI / откат / экспорт / цена».
- [ ] Цены — открыто, без подписок. «100₽ стартовый баланс, дальше за токены, всё в кошельке».
- [ ] Страница «Шаблоны» с превью каждого шаблона.
- [ ] **Блог** с long-tail SEO-статьями: «как сделать лендинг для X», «AI vs no-code в 2026», «как запустить сайт за 1 день в РФ». Цель — Phase B.

### 2.3. Phase B — сгенерированные сайты (после первых платных юзеров, ~3 дня)

Каждый сайт, сгенерированный AI, должен быть **SEO-готовым из коробки**. Это
*продуктовая* фича, не маркетинговая.

**Что зашиваем в системный промпт LLM** (`apps/api/src/omnia_api/services/prompt_builder.py`):

```
Каждая сгенерированная HTML-страница ОБЯЗАНА содержать:
- <title> длиной 50–60 символов
- <meta name="description"> 140–160 символов
- <meta property="og:title">, <meta property="og:description">, <meta property="og:image">
- <link rel="canonical" href="...">
- <html lang="ru"> (или язык, определённый по промпту)
- Семантические H1/H2/H3 с ключевыми словами из промпта
- alt="..." для каждой <img>
```

**Что генерим автоматически** (api-side after generation):

- `sitemap.xml` для каждого проекта (`/p/<slug>/sitemap.xml`)
- `robots.txt` для каждого проекта (`/p/<slug>/robots.txt`)
- JSON-LD `LocalBusiness` / `Product` / `Organization` — детектим по типу сайта, прописываем в head

**Per-snapshot canonical**: текущий HEAD-snapshot — canonical, исторические — `noindex`.

### 2.4. Phase C — производительность и Core Web Vitals (~2 дня)

- Все сгенерированные сайты деплоятся как **статика** (HTML/CSS/JS без SSR где возможно).
- CDN (Cloudflare или Selectel CDN) перед генерёнными сайтами.
- Image optimization — даже если AI сгенерил `<img src="big.png">`, наш билд-степ перегоняет в WebP/AVIF.
- Preload критических шрифтов.
- Targets: LCP < 2.5s, CLS < 0.1, INP < 200ms.

### 2.5. Phase D — рост по сигналам (постоянно)

- Отзывы пользователей с разрешением публиковать.
- Backlinks через каталоги: vc.ru, Хабр, Клерк (для бизнес-аудитории), Pikabu для виральности.
- Каждый сгенерированный сайт пользователя → **footer** «Сделано на Omnia.AI» (опционально, отключаемо за 199₽/мес) → бэклинк-эффект.
- Webhook на новые регистрации в Yandex Webmaster — сразу запрос индексации.

## 3. Домены — план

### 3.1. Стратегия: ресселл REG.RU

Не становимся регистратором сами (это ФЗ-149, лицензия ICANN, аккредитация
Координационного центра .RU — годы переговоров). Становимся **ресселером**.

| Партнёр | Что даёт | Что забирает |
|---|---|---|
| **REG.RU Reseller** | API на регистрацию, продление, перенос; whitelabel'ный WHOIS на их юрлицо; абсолютно вся юридическая нагрузка | оптовая цена (.ru ~169₽/год оптом vs 199₽ розница), мы продаём по 249₽ — маржа ~30% |
| Beget Partner | Аналогично | Чуть дороже оптом, но проще API |
| timeweb | Аналогично | Меньше выбор зон |

**Решение для MVP: REG.RU**. Самая большая ассортиментная база (.ru/.рф/.com/.online/.shop/etc), стабильный API, есть юрлицо-обёртка.

### 3.2. Что нужно от REG.RU

1. Партнёрский договор — **оффлайн**, ~2 недели подписания.
2. Логин/пароль для API + URL `https://api.reg.ru/api/regru2/`.
3. Тестовый аккаунт для прогона до реальных оплат.

### 3.3. API-flow

```
[Юзер кликает «купить домен»]
   ↓
omnia/web → omnia/api: POST /api/domains/check {name}
   ↓
omnia/api → REG.RU: domain/check {domains: ["coffee-cloud.ru"]}
   ↓
REG.RU: {available: true, price: 249.00 RUB}
   ↓
[Юзер видит цену + жмёт «Оформить»]
   ↓
[Если passport_data НЕТ — показываем форму один раз]
[Если уже есть — пропускаем шаг]
   ↓
omnia/api → omnia.wallet: списать 249 RUB (создаём pending charge)
   ↓
omnia/api → REG.RU: domain/create_user (создаём контакт-юзера в REG.RU
                  с нашими паспортными данными — для ".ru" обязательно)
   ↓
omnia/api → REG.RU: domain/create {domain, contacts, ns: omnia-ns1, omnia-ns2}
   ↓
REG.RU: {result: success, domain_id: ...}
   ↓
omnia/api: INSERT domains (user_id, name, status='active', expiry, regru_id)
omnia/api: INSERT wallet_charges (settled)
omnia/api: попросить worker:
   - DNS A-record → 170.168.72.200 (наш VPS) или CNAME → constructor.lead-generator.ru
   - certbot --nginx -d <new-domain> (Let's Encrypt автоматический)
   - nginx server-block из шаблона
   ↓
omnia/api → user (websocket): "domain.ready"
```

Перевыпуск SSL — cron, который у нас уже есть от certbot. Новый сертификат —
просто новый запрос `certbot certonly`.

### 3.4. Хранение паспортных данных (152-ФЗ)

- В отдельной таблице `merchant_profiles` — **encrypted at rest** через `pgcrypto` (AES-256, ключ в env `PII_ENCRYPTION_KEY`, хранится в k/v, **не** в git).
- Доступ к расшифровке — только сервису `services/registrar/` через явный flag.
- **Согласие 152-ФЗ** — отдельная галочка в UI с фиксацией `consented_at` и текстом версии.
- **Право на забвение** — endpoint `DELETE /api/merchant/profile` зануляет поля и пишет audit log.
- **Журнал доступа** — каждый раз когда мы декриптим запись для отправки в REG.RU, пишем в `pii_access_log`.

### 3.5. Цены и юнит-экономика

| Зона | Опт REG.RU | Наша цена | Маржа |
|---|---:|---:|---:|
| .ru | 169 | 249 | 80₽ (47%) |
| .рф | 169 | 249 | 80₽ |
| .com | 850 | 1090 | 240₽ (28%) |
| .online | 1499 | 1990 | 491₽ (33%) |
| .shop | 1799 | 2390 | 591₽ |
| **продление** | те же | те же ×1.0 (без наценки) | retain |

## 4. Юкасса — план

### 4.1. Стратегия: маркетплейс через ЮKassa Connect

Юкасса умеет работать в **режиме платформы**: ты юрлицо-платформа, твои
пользователи — sub-merchants, деньги идут от их клиентов через тебя к ним
(минус твоя комиссия). Это называется **«Маркетплейс»** или **«Эквайринг для
платформ»**.

Альтернатива: каждый юзер сам ходит в Юкассу и подписывает договор.
*Плохо* — высокий churn из-за бюрократии.

**Решение: ЮKassa Connect (платформа)**. Мы делаем 1 договор, юзеры
подключаются через onboarding API за минуты.

### 4.2. Что нужно от ЮKassa

1. Регистрация юрлица **Omnia** (пока не существует — см. §6).
2. Договор «маркетплейс» с Юкассой — минимум 2 недели + due diligence.
3. shopId + secretKey для API.
4. Подключение «Самозанятым» / «ИП» / «ООО» через их Connect API.

### 4.3. Onboarding flow (для физика, желающего принимать платежи)

```
[Юзер в Omnia: «хочу принимать оплату на сайте»]
   ↓
omnia: проверяем merchant_profile — есть ли паспорт
   ├── нет → показываем большую форму PII (один раз!)
   └── есть → пропускаем
   ↓
omnia: проверяем business_profile — есть ли ИНН/ОГРН
   ├── нет → форма «выберите статус»:
   │           — Самозанятый (нужен только ИНН)
   │           — ИП (нужны ОГРНИП, реквизиты счёта)
   │           — ООО (полный пакет)
   └── есть → пропускаем
   ↓
omnia → ЮKassa Connect API: POST /partners/me/sub-accounts {
   passport_data, inn, ogrn, bank_account, ...
}
   ↓
ЮKassa: статус — обычно «pending», 1–3 рабочих дня модерации.
   ↓
omnia: записываем yookassa_account_id, yookassa_status='pending'
omnia: показываем юзеру: «Заявка отправлена. Активируем в течение 1–3 дней,
       позовём в Telegram/email»
   ↓
[через 1–3 дня] ЮKassa webhook: «account.activated»
   ↓
omnia: yookassa_status='active', шлём email + WS-event
omnia: автоматически добавляем в каждый сайт юзера платёжный виджет
       (если он включил «Принимать оплату» в проекте)
```

### 4.4. Поток денег

```
[покупатель на сайте юзера] → ЮKassa виджет → 1000₽
   ↓
ЮKassa: фиксирует платёж под маркетплейсным shopId с saldo split:
   - 970₽ → юзер (sub-merchant)
   - 25₽ → Юкасса (комиссия эквайринга 2.5%)
   - 5₽ → omnia (наша комиссия 0.5%)
```

ЮKassa сама выводит юзерским sub-аккаунтам по их реквизитам по графику.
Мы ничего не трогаем.

### 4.5. Цены

- **Эквайринг (для покупателя на сайте юзера):** ~2.8% (ЮKassa + наши 0.3% сверху).
- **Подключение к маркетплейсу:** бесплатно.
- **Минимальный остаток на счёте omnia-юзера**: 0₽ (мы не блокируем).

## 5. Единый onboarding — «один раз заполнил, везде работает»

Это ключевая UX-идея. Реализуем в виде **сквозного wizard**:

### 5.1. Маршрут

```
Регистрация → /projects (бесплатный режим) →
   юзер делает сайт →
   жмёт «Опубликовать на своём домене» ИЛИ «Принимать оплату» →
   попадает в /onboarding →
   проходит шаги последовательно
```

### 5.2. Wizard-шаги

| # | Шаг | Что заполняем | Откуда переиспользуется |
|---|---|---|---|
| 1 | Тип субъекта | физлицо / самозанятый / ИП / ООО | определяет ветку |
| 2 | Паспортные данные | ФИО, серия+номер, кем выдан, когда, регистрация, дата рождения | для домена + ЮKassa |
| 3 | Контакты | телефон, дополнительный email | для домена + ЮKassa |
| 4 | (если ИП/ООО) Бизнес-данные | ИНН, ОГРН/ОГРНИП, юрадрес, расчётный счёт + БИК | для ЮKassa |
| 5 | Согласия | 152-ФЗ, оферта Omnia, оферта ЮKassa, оферта REG.RU | юридически обязательно |
| 6 | Покупка домена (опционально) | имя домена + зона + срок | префилл из (2)–(4) |
| 7 | Подача в ЮKassa (опционально) | автозаполнено из (2)–(4) | без повторного ввода |
| 8 | Финиш | статусы каждого шага в реальном времени | домен ~5 мин, ЮKassa ~1–3 дня |

### 5.3. Состояние и идемпотентность

- `merchant_profiles` — *одна строка на пользователя*, обновляется при каждом
  редактировании.
- Заявки в REG.RU/ЮKassa — отдельные таблицы `domains`, `yookassa_accounts`
  с `external_id` для идемпотентности (повтор клика «отправить» не делает
  второй платёж).
- Webhook от REG.RU и ЮKassa — отдельная таблица `webhook_events` с дедупом
  по `event_id` (no double processing).

## 9. Что точно НЕ делаем в MVP

- ❌ Авто-обновление контента «после публикации» (это уже отдельный продукт)

# Явный бэкенд: реальные таблицы из сущностей — дизайн

**Дата:** 2026-06-08
**Статус:** одобрен к спецификации (ждёт вычитки владельцем)
**Область:** шаблон `apps/orchestrator/templates/nextjs-entities/` + оркестратор-хук `hot_reload`

---

## Северная звезда (видение)

Долгосрочная цель — воркфлоу, где можно собрать **что угодно, без потолка**. Этот кирпич — фундамент, который сознательно **не ограничивает** общность:

- Генерёное приложение остаётся **обычным Next.js + реальный Postgres** — туда можно дописать произвольный код, таблицы и эндпойнты рядом с движком сущностей.
- «Без ограничений» достигается **слоями escape-hatch'ей ПОВЕРХ** надёжного ядра (кастомная бизнес-логика, произвольные стеки, прямой SQL) — отдельными под-проектами, а не размыванием этого.
- **Принцип: открытая система, не закрытая.** Движок сущностей покрывает типовые 80% (данные/CRUD/вход/связи); остальное всегда доступно через прямой код. Реальные таблицы (vs прежний один JSONB) — шаг ИМЕННО в сторону общности.

Этот спек реализует ядро; escape-hatch'и перечислены в «Вне scope / отложено».

## Цель и scope

`entities/<Name>.json` перестаёт жить в одной generic-таблице `records (… data JSONB …)`. Каждая сущность компилируется в **свою реальную таблицу** Postgres: типизированные колонки, индексы, внешние ключи по связям. Декларацию (entity JSON) по-прежнему пишет AI; новый слой **реконсилирует** её в реальную схему. Касается только стека `nextjs_entities`.

## Ключевые решения (подтверждены владельцем)

1. **Реальные таблицы из сущностей** — не «AI пишет весь бэкенд» и не гибрид (пока). Движок остаётся, но материализует настоящую схему.
2. **Аддитивно-безопасная эволюция** — авто-применяем только добавления (новая таблица/колонка/индекс). Разрушающее (удаление колонки, смена типа) НЕ авто — копится в отчёт и применяется явным шагом. Данные не теряются никогда.
3. **Гибрид материализации** — реконсилятор делает DDL чистым SQL (надёжно, без drizzle-kit-ханга), плюс кодоген генерит `schema.ts` (Drizzle) как витрину типов и «явный» артефакт для dev/экспорта.

### Почему не альтернативы
- `drizzle-kit push` **виснет** на общем per-project Postgres (ради этого и был написан `init-db.mjs`) и не аддитивно-безопасен → не годится как источник DDL.
- «AI пишет бэкенд» — менее надёжно (модель лажает в серверном коде) и дороже; оставлено на escape-hatch-слой.

## Архитектура / компоненты

Изолированные модули, каждый с одной задачей:

1. **`src/lib/db/typemap.ts`** — отображение «тип поля → колонка Postgres»:
   `string→text`, `text→text`, `number→numeric`, `boolean→boolean`, `date→timestamptz`,
   `enum→text + CHECK (val IN (...))`, `reference→uuid + FK → <target>.id`.
   Плюс служебные колонки каждой таблицы: `id uuid PK default gen_random_uuid()`, `created_by uuid → users.id`, `created_at timestamptz`, `updated_at timestamptz`.
   (enum как `text+CHECK`, не pg-enum — проще мигрировать; `number` как `numeric` — безопасно для денег.)

2. **`src/lib/db/reconcile.ts`** — ядро. Вход: разобранные `entities/*.json` + интроспект текущей БД (`information_schema`). Считает дифф и применяет **только аддитив** чистым `pg`: `CREATE TABLE IF NOT EXISTS`, `ALTER TABLE … ADD COLUMN IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`, `ADD CONSTRAINT` (FK) идемпотентно. Разрушающие диффы (колонка исчезла из декларации, сменился тип) — НЕ применяет, складывает в отчёт `pending_destructive` (лог + endpoint для будущей явной кнопки). Всё в одной транзакции.

3. **`src/lib/db/codegen-schema.ts`** — генерит `src/lib/db/schema.ts` (Drizzle-таблицы) из тех же entity JSON. Это **витрина типов + «явный» артефакт** (его читает dev, он уезжает в экспорт), НЕ источник DDL (источник — реконсилятор). Auth-таблицы (`users/accounts/sessions/verificationTokens`) остаются как есть.

4. **`src/lib/entities/engine.ts`** — переписать CRUD с generic-`records` на **per-entity таблицы**: имя таблицы = `snake_case(entity)`, колонки = поля. Owner-scoping через колонку `created_by`. Expand связей — реальный FK (батч-запрос или JOIN). Имена таблиц/колонок берутся из реестра (валидируются как безопасные идентификаторы — регэксп уже есть в `registry.ts`), значения параметризуются.

5. **`scripts/init-db.mjs`** — на буте вызывает реконсилятор (вместо создания фикс-`records`). Сохраняет существующий fail-soft (30s timeout, старт даже при сбое БД).

6. **Оркестратор `hot_reload`** (`apps/orchestrator/.../runtime.py`) — при изменении `entities/*.json` гоняет реконсилятор внутри контейнера (на месте прежнего вызова `drizzle-kit push`, но это наш скрипт — без ханга).

## Поток данных

```
AI пишет/правит entities/*.json
   → reconcile.ts: интроспект БД + аддитивный дифф → DDL (реальные таблицы)
   → codegen-schema.ts: обновляет src/lib/db/schema.ts (типы/явность)
   → engine.ts: CRUD по реальным per-entity таблицам (owner-scope, FK-expand)
Разрушающие правки → отчёт pending_destructive (применяются явной командой, см. отложено)
```

## Откат и совместимость

- За флагом **`ENTITY_REAL_TABLES`**. Новые `nextjs_entities`-аппы — реальные таблицы по умолчанию.
- Существующие JSONB-аппы (магазин помады и пр.) продолжают работать на `records` без изменений (движок выбирает режим по наличию флага/маркера в проекте).
- Одноразовый мигратор `records → per-entity tables` — по желанию, не обязателен. Никаких ломающих изменений у живых аппов.

## Обработка ошибок / fail-safe (R-10)

- Реконсилятор **только аддитивный** → структурно не может потерять данные.
- DDL в одной транзакции; `IF NOT EXISTS` делает повторный прогон идемпотентным.
- Сбой реконсилятора → контейнер всё равно стартует (fail-soft, как `init-db.mjs` сейчас); CRUD по отсутствующей таблице → понятная 4xx-ошибка через `http.ts`, не краш процесса.

## Тесты

- **Юнит:** `typemap` (каждый тип→колонка + служебные); дифф (аддитив vs разрушающее — классификация); `codegen-schema` (entity JSON → валидный Drizzle).
- **Интеграция:** entity JSON → реальная таблица с индексами + FK; добавил поле → `ADD COLUMN`, прежние строки целы; удалил поле → попало в отчёт, колонка не тронута; reference → реальный FK + expand работает.
- **E2E:** новый entity-апп — полный CRUD по реальным таблицам + связи (зеркало проверки магазина помады, но на реальных таблицах).

## Вне scope (YAGNI) / отложено отдельными под-проектами

- Авто-применение разрушающих правок и детект переименований (rename vs add+drop).
- Массовая миграция всех старых JSONB-аппов разом.
- **Escape-hatch-слои к «без ограничений»:** гибрид (AI дописывает кастомные эндпойнты/логику поверх движка), произвольные стеки, прямой SQL-доступ.
- Авто-роутинг стека (промпт сам понимает «нужен бэкенд»).
- Хороший дизайн app-UI (таблицы/формы/дашборды) — параллельный трек.
- RBAC/команды, rate-limit, аудит.
- Live-рендер фронта + геймификация.

## Открытые вопросы

Блокирующих нет. Мелкие решения зафиксированы инлайн: `enum→text+CHECK`, `number→numeric`, имена таблиц = `snake_case(entity)`.

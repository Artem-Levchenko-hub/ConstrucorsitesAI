## 1. Спринт +1: «Превью — живой инструмент» (1–2 дня)

### 1.1. Hot reload при новом снэпшоте

**Сейчас:** при появлении `snapshot.created` event'а в timeline появляется новая карточка, но iframe остаётся на старой версии. Юзер должен кликнуть на неё.

**Хочется:** новая снэпшота — iframe сам плавно переключается на неё.

**Как:** в `usePromptStream.ts` на `snapshot.created` event автодиспатчить `selectSnapshot(event.data.snapshot.id)`. Анимация 200мс fade-out → fade-in, чтобы это выглядело как «обновление».

### 1.2. Стримящееся превью

**Сейчас:** превью обновляется только когда полный снэпшот готов (10–20 сек).

**Хочется:** видеть как сайт **строится** в реальном времени, как Claude Artifacts.

**Как:**
- В чате уже видно потоковый ответ AI (через WS llm.chunk).
- Параллельно api копит chunks → как только у нас есть валидный HTML (закрыт `</body>`), парсит и складывает временный файл в Redis.
- Iframe слушает свой собственный WS-канал (`/api/ws/projects/{id}/preview`) и при получении нового HTML делает `iframe.contentDocument.write(html)`.
- Альтернатива безопасней: эндпоинт `/p/<slug>?stream=draft` отдаёт draft-версию из Redis (живёт минуту).

**Тоговый UX:** пишешь промпт → правая половина превью на твоих глазах строится сверху-вниз.

### 1.3. Click-to-edit элементы

**Сейчас:** хочешь поменять заголовок — пиши промпт «поменяй текст hero на ...».

**Хочется:** клик на элемент в превью → попап «отредактировать промптом», текущий текст уже в инпуте, отправка сразу даёт промпт «измени `<h1>Кофейня</h1>` на ...».

**Как:**
- Iframe (same-origin) → можем инжектить overlay-script.
- Скрипт ловит `mouseover` → выделяет элемент рамкой, `click` → отправляет в parent `postMessage({type: "edit-element", html, selector, text})`.
- Workspace показывает inline-input у элемента, при submit вызывает `usePromptStream.submit("Замени `<селектор>` ...")`.
- Это **главная фишка**, отделяющая нас от Tilda и no-code.

### 1.4. Keyboard shortcuts

| Hotkey | Действие |
|---|---|
| `Ctrl+Enter` в чате | Отправить промпт (мини-инвестиция в скорость) |
| `R` | Reload iframe |
| `1` / `2` / `3` | Mobile / Tablet / Desktop |
| `Z` (без Ctrl) | Откатить на предыдущий snapshot |
| `K` | Открыть command palette с моделями + действиями |
| `?` | Показать список горячих клавиш |

### 1.5. Diff между снэпшотами

При наведении на снэпшот в timeline (или клик «сравнить» рядом с откатить) — модалка со split-view: «было / стало», подсвеченные изменения файлов. Используем существующий `<file path="">` парсинг + diff-библиотеку (`diff` npm, на клиенте).

## 2. Спринт +2: «AI помогает не остановиться» (2–3 дня)

### 2.1. Авто-подсказки следующих промптов

**После каждого generation** AI генерит ещё 3 коротких подсказки в стиле «может быть, дальше?» — кнопки прямо под чатом:
- «Добавить раздел отзывов»
- «Сделать форму обратной связи в модалке»
- «Подключить Yandex.Maps к секции «контакты»»

Реализация: после ответа модели делаем второй короткий запрос к самой дешёвой модели (gigachat-2 за 0.20 ₽/1k) с system-prompt'ом «Предложи 3 next-step'а одной строкой каждое».

### 2.2. Автогенерация мета-тегов и SEO (Phase B SEO)

В системный промпт добавляем требование выдавать:
- `<title>` 50–60 chars
- `<meta name="description">` 140–160 chars
- `<link rel="canonical">`
- OG-теги
- JSON-LD `LocalBusiness` или `Product` (детектим по контексту)
- `<html lang>` правильный
- `alt` на всех `<img>`

Уже описано в `docs/04-monetization-plan.md` Phase B. Сейчас именно сюда.

### 2.3. Авто-Lighthouse для каждого снэпшота

Worker рендерит preview через Playwright. Добавим к этому:
- Запуск Lighthouse в Playwright (`page.lighthouse()`)
- Сохраняем 4 числа: Performance / SEO / Accessibility / Best Practices
- В timeline под снэпшотом — мини-бейджи `S 95 P 87 A 100`
- Если что-то < 70 — красная подсветка + кнопка «Попросить AI починить»

### 2.4. Автодеплой + custom domain

Уже есть на `/p/<slug>`. Дальше:
- Юзер прикрепляет свой домен (через REG.RU из плана 04)
- Мы автоматически: A-record → 170.168.72.200, certbot --nginx -d, добавляем в `domains` таблицу.
- За кадром обновляется nginx server-block; HEAD сразу доступен по https://их-домен.

## 3. Спринт +3: «Весь cycle жизни проекта на сайте» (3–5 дней)

### 3.1. Code editor inline

Кнопка «Показать код» — справа от превью открывается Monaco editor с текущим HTML/CSS/JS. Изменения сохраняются как новый snapshot (с пометкой `manual_edit=true`). Промпты и manual edits живут в одном linear timeline.

### 3.2. Asset library

- Drag&drop картинки в превью → загружаются в MinIO-бакет `assets/{project_id}/`, путь автоматически вставляется в html.
- Сторонние изображения с `<img>` (например, AI выдумал `https://example.com/coffee.jpg`) — фоновая job заменяет на наш asset через unsplash search или generate (DALL-E через gateway, опционально).

### 3.3. Templates marketplace

- Юзер может **опубликовать** свой проект как шаблон. Мы делаем форк-снэпшот.
- Другой юзер выбирает «Старт с шаблона X» — клонируется в его репозиторий.
- Мотивация для авторов: revenue-share с топов (опционально, после первой партии).

### 3.4. Built-in аналитика

После публикации каждый сайт получает наш миниатюрный JS-снифер (5KB):
- Количество визитов
- Среднее время на странице
- Откуда пришли (referrer, UTM)
- Какие кнопки кликали (на основе `data-track=...` атрибутов которые AI добавляет автоматически)
- Все данные — в Postgres `site_analytics`, дашборд для юзера в workspace.

Привлекает к нашему сайту еще больше — все маркетинговые цифры тут же.

### 3.5. AI-help on hover

Юзер навёл на любой блок дольше 3 сек → всплывает «🔍 Объяснить почему так?» — клик, AI пишет «потому что выбрана палитра…, шрифт Inter подобран потому что…, кнопка центрирована потому что…». Учит юзера дизайну ненавязчиво. Запрос дёшев, gigachat-2 хватит.

## 4. Cпринт +4: «Командная работа» (1–2 недели)

### 4.1. Шаринг проекта без аккаунта

Кнопка «Поделиться» рядом с «Открыть» → генерирует ссылку `/p/<slug>?ro=<token>` (read-only). Кто угодно с ссылкой — видит превью, чат, timeline. Даже без аккаунта. Может оставлять комментарии, требует логина.

### 4.2. Многопользовательские проекты

Owner может пригласить collaborator'а (email). Тот логинится, видит все проекты, к которым приглашён. Оба могут отправлять промпты, видят чат друг друга. Race на `current_snapshot_id` решается оптимистическим conflict — последний выигрывает, но в timeline видно ветку.

### 4.3. Комментарии на элементах

Click-to-edit (1.3) можно расширить до click-to-comment: «эта кнопка должна быть зелёной» оставляется как сноска. Когда юзер потом отправляет промпт «исправь все мои комменты» — AI читает комменты + текущий код + одним промптом фиксит всё.

## 6. Технические инвенстиции (фон)

### 6.1. WebSocket reconnect & state sync

Сейчас WS открывается на mount, при разрыве не реконнектится. Нужно:
- Reconnect с exponential backoff
- При коннекте вытягивать `?since=<last_event_id>` чтобы получить пропущенные события
- В api добавить event-store для последних 100 событий per-project в Redis stream

### 6.2. Optimistic UI

При отправке промпта **сразу** показывать в чате user-сообщение и пустой assistant с loader'ом — не ждать 202 от api. Если 4xx — откатить с error toast.

### 6.3. Server-side preview cache

`/p/<slug>` сейчас при каждом hit идёт в pygit2 + MinIO. Кэшировать в Redis на 60 сек (с инвалидацией при snapshot.created).

### 6.4. Logging & traces

Один `request_id` от первого хитa в web → api → gateway → возврат, в каждом docker-логе видно цепочку. Пока структурно нет, страдает дебаг.

### 6.5. Тесты

Playwright e2e для основных user-flow'ов:
- регистрация → логин → создание проекта
- промпт → snapshot → preview обновился
- откат → старая версия в iframe
- click-to-edit → новый промпт ушёл

Запуск в CI на каждый PR. Сейчас — ноль таких тестов.

## 8. Что НЕ делаем

- ❌ Свой Visual page builder (drag&drop блоки) — вступает в конфликт с AI-промптами, путает юзера.
- ❌ Собственный CMS интерфейс (страницы, посты, типы контента) — это другой продукт. Если AI генерит нужные страницы — этого достаточно.
- ❌ Многоязычные интерфейсы UI — пока только русский. На сгенерированных сайтах — любой, AI сам определяет.
- ❌ Plugin marketplace — слишком ранний этап для extension API.

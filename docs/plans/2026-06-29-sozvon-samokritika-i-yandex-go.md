## Как держать рамку (30 секунд в начале)

> «У нас **не прототип, который надо переписать. У нас рабочий двигатель**, которому нужны ремни безопасности и пара подключённых проводов — и мы знаем каждый по имени, файлу и цене.»

Три правила честной подачи:
1. Каждый минус называем **вместе с тем, что лекарство уже написано** — лежит за флагом или одной строкой. Проблема — *подключить*, а не *изобрести*.
2. Чётко разделяем два разных «осталось»: **наши недоделки обвязки (часы-дни) ≠ потолок технологии 2026 (недели-месяцы / принципиально)**. Не смешивать.
3. На вопрос «потянем ли Яндекс Go» отвечаем прямо (ниже), без приукрашивания.

## ГЛАВНЫЙ ВОПРОС: потянем ли уровень «Яндекс Go» языковыми моделями?

**Прямой ответ: целиком — НЕТ. Это не «дозреет за пару лет» и не наша слабость — это потолок технологии.**

«Яндекс Go» — не «приложение», а распределённая система национального масштаба: ~10–15 лет работы тысяч инженеров, сотни сервисов под миллионы запросов/сек, реал-тайм диспетчеризация за миллисекунды, свой ML (время подачи, спрос, антифрод), карты и маршруты, платёжная инфраструктура с лицензиями, нативные приложения в сторах, надёжность 99.99% (~52 минуты простоя в ГОД).

Языковая модель в 2026 (и наш агент) — **сильный инструмент кодирования отдельных кусков под надзором инженера**, а не субъект, который держит в голове и развивает систему такого масштаба.

| Часть «Яндекс Go» | ИИ-агент? | Почему |
|---|---|---|
| Поддержка 24/7: тикеты, панели, боты | 🟢 под надзором | то, что наш билдер уже умеет (CRUD, админки, чат, роли) |
| Ценообразование (правила) | 🟡 частично | движок по правилам — да; «правильная» экономика = ДАННЫЕ |
| Отдельные сервисы (API+база+логика) | 🟡 поштучно | наш хлеб; но **сборка сотен в надёжное целое** — нет |
| Мобильные приложения (сторы) | 🔴 нет в обозримом | нет мобильных стеков; публикация = ручное ревью, подписи, SDK |
| ML времени/спроса | 🔴 нет | качество = ДАННЫЕ + эксперименты, не «написать код» |
| Реал-тайм матчинг (ядро) | 🔴 принципиально | корректность распределёнки под живой нагрузкой; модель не проверяет |
| Платежи + антифрод | 🔴 принципиально | лицензии, PCI DSS, юр-ответственность + гонка с противником |
| Масштаб (миллионы rps) | 🔴 нет | настройка против ЖИВОГО трафика, которого модель не видит |
| Надёжность 99.99% / эксплуатация | 🔴 принципиально | дежурства, постмортемы, бэкапы — дисциплина организации |

**Ограничение — не «модель слабовата».** У LLM нет: устойчивого долгосрочного состояния всей системы, ответственности перед законом/деньгами/жизнями, проверки под реальной нагрузкой, доступа к правовым/банковским/платформенным контурам. Это не открывается ростом модели.

**Роль ИИ на 3–5 лет:** строит 60–80% **рутинного** кода такого класса ВНУТРИ команды и резко поднимает её скорость. ИИ **не заменяет** организацию и не «выдаёт Яндекс Go по промпту».

## ЧТО РЕАЛЬНО ПО СИЛАМ (наш достижимый рынок)

> Честная формула: **агент строит ПРОДУКТ (фичу, кабинет, модуль), но не ПЛАТФОРМУ (изоляция данных, эксплуатация, масштаб, много клиентов, разнородные клиенты).**

- ✅ **B2B-кабинет / отраслевая CRM на один бизнес** (стоматология: кабинет+запись+роли+админка для ОДНОЙ сети) — **сейчас**, после 3 фиксов ниже.
- ✅ **Внутренняя корпоративная система** (заявки, инвентарь, расписание, согласования) — самый сильный класс: нет публичной нагрузки, один арендатор, проверяемо нашими глазами.
- ✅ **Realtime ограниченного масштаба** (чат/лента/коллаб) — уже подключён, единственный стек с блокирующей проверкой поведения.
- 🟡 **Ранний SaaS (MVP → первые платящие)** — сейчас→6 мес, упирается в изоляцию данных и эксплуатацию.
- 🟡 **Telegram-сервис с платежами / серверный API** — шаблоны собраны, не хватает «проводки» в роутер (недели).
- 🟡 **Маркетплейс/ERP-модуль среднего размера** — 1–2 года.

## САМОКРИТИКА: почему ещё не работает надёжно и что сделать

### А. Наши недоделки обвязки (часы–дни) — лекарство уже написано
1. **Инструмент «свежие доки» — мёртв (одна строка!).** Прорекламировали модели «подтяни актуальные доки вместо угадывания», написали обработчик — но забыли добавить слово в список разрешённых действий. Каждый вызов отвергается → модель угадывает API по памяти → лишние ошибки сборки. *Фикс: добавить в список + тест. Часы.*
2. **Главный рычаг надёжности правок — не подключён.** «Прощающее» сравнение (прощает пробел/отступ) написано, но в старой ветке. Живой редактор требует копировать байт-в-байт → дешёвая модель мажет → правка тихо не применяется. *Фикс: подключить готовое. Часы. Эффект ×кратный.*
3. **Усиление модели теряется на продолжениях.** Большие апп собираем за несколько заходов; «повышение» до сильной модели из захода 1 не передаётся дальше — трудный билд снова на дешёвой. *Фикс: пробросить состояние. Часы.*

### Б. Серьёзные пробелы продукта (недели)
4. **Приложения с базой шипятся без блокирующей проверки.** Гейт «работает и не течёт» — только для чатов, и даже там при провале лишь дописывает предупреждение и всё равно коммитит. Проверки ролей/безопасности написаны и протестированы, но **в бою не вызываются — мёртвый код**, хотя галочки стоят. *Фикс: провал = блок + на все стеки; мёртвые гейты подключить или честно выключить. Недели.*
5. **Изоляция данных между клиентами в генерируемых апп — нет.** Есть колонка «чей объект», нет защиты на уровне базы. «Одна сеть» — да, «коробка на сотни клиник» — нет. *Фикс: RLS + скоуп + проверка в гейте. Недели.*
6. **Стеки: API и телеграм-бот собраны, но роутер их не знает; мобайл/Go/Rust/.NET — нет вообще.** *Фикс: api/tgbot в роутер (дни); дальше generic-провижн + языки (недели/стек); мобайл — отдельный трек.*

### В. Безопасность платформы (дни, но БЛОКЕР широкого запуска)
7. **Песочница не изолирована по умолчанию.** Агент гоняет произвольные команды в контейнере проекта — а контейнеры в общей сети, без ограничения исходящего трафика, под обычным runc; есть вектор утечки общего ключа от хранилища. Механизмы изоляции написаны и безопасны — просто выключены (нужна подготовка хоста). *Фикс: подготовить хост + 4 настройки. Дни.*

> (Платежи/бэкапы/CI/метрики — по твоей просьбе выносим за скобки; это отдельный трек «зрелость сервиса».)

## Финальная формула для созвона

> «Движок настоящий и работает — мы видим свои ошибки и сами чиним сборку, это сильнее конкурентов-одношотеров. Но честно: надёжность ещё не продакшн — правки срываются на дешёвой модели, серьёзные апп едут без блокирующей проверки, песочница не изолирована. Это **решаемая инженерная работа: часть — часы, часть — недели**, и мы знаем каждый пункт по имени, файлу и цене. А продукт масштаба Яндекс Go языковыми моделями целиком не строится — ни у нас, ни у кого, это потолок технологии. Зато реальные вертикали под ключ — кабинет клиники для сети, внутренние системы, ранний SaaS — это наш достижимый рынок уже этого года.»

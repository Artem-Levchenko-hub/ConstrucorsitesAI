# Dogfood-Eval Routine — протокол (источник правды)

**Создан:** 2026-06-16. **Запуск:** scheduled-task `dogfood-eval-loop`, cron `*/10 * * * *` (каждые 10 мин), последовательно, уступает живым сессиям. Токены не ограничены.

## Миссия

Автономно догфудить ПРОДУКТ (constructor.lead-generator.ru) как живой пользователь: генерить приложения / скрипты / ботов, оценивать **дизайн И функциональность** по скриншотам + логам, **находить тупняки и слепые зоны** генератора, и **чинить их по доказательствам (логи + скриншоты), НЕ вслепую**. Каждый прогон — один цикл, evidence → diagnosis → (fix или proposal) → ledger.

## Главный известный тупняк (гипотеза H1 — проверить ПЕРВОЙ)

Владелец: «написал "создай сайт" → получил статику; кинул 5 промптов "переделай в полноценное веб-приложение" — генератор по сути ничего не делал, чуть перегенривал статику, не отвечал по запросу».

**Гипотеза причины** (из root-cause анализа пайплайна):
- Первый промпт → `discovery` выбирает stack; дефолт `static` (`discovery._DEFAULT_STACK`), а полноценный апп — только если LLM-рекомендер активно выберет контейнер.
- **Follow-up промпты идут через triage → surgical-edit / edit-путь, который правит СУЩЕСТВУЮЩИЙ static HTML и НИКОГДА не пере-запускает discovery / stack_routing.** То есть «переделай в приложение» трактуется как правка статики, а не как смена стека static→nextjs_entities/spa. Эскалации стека на follow-up НЕТ.
- Проверить: `apps/api/src/omnia_api/routers/messages.py` (triage/intent → surgical/edit ветка), `services/intent_triage.py`, `services/discovery.py` (запускается ли на follow-up), `services/stack_routing.py` (`switch_to_stack` вызывается ли вне первого билда).
- **Фикс-направление** (когда подтверждено логами): detect intent «сделай настоящее приложение / добавь вход / БД / кабинет» на follow-up → re-trigger discovery/stack-escalation (static→container), а не surgical-edit статики. С тестом-репро.

## Протокол одного прогона (строго по шагам)

### 0. ГАРД (если не прошёл — выйти, записать строку в LEDGER «skipped: <причина>»)
- **Sequential-lock:** файл `_routine/dogfood.lock`. Если есть и моложе 15 мин → другой прогон идёт → ВЫЙТИ. Иначе записать в него ISO-время старта.
- **Уступи живой сессии:** glob `C:\Users\79133\.claude\coordination\omnia-mvp-*\active-sessions.json`. Если есть сессия с heartbeat за последние 5 мин и это НЕ этот прогон → ВЫЙТИ («yielded to active session»). Нет файла/дир → считать idle, продолжать.
- **Resource-guard:** `ssh i48ptgvnis@170.168.72.200 'free -m'`. Если available < 3000 МБ → НЕ генерировать: `docker rm -f` висяки `omnia-dev-{routine,test,dogfood}-*`, перепроверить; всё ещё мало → ВЫЙТИ.
- Снять lock в КОНЦЕ прогона всегда (даже при ошибке).

### 1. ВЫБОР СЦЕНАРИЯ (ротация по номеру прогона = число строк RUNS в LEDGER mod N)
0. **H1-репро (приоритет, пока H1 открыт):** «создай сайт для кофейни» → дождаться → затем «переделай это в полноценное веб-приложение: вход, личный кабинет, база записей» → ещё 1-2 таких. Замерить: эскалировал ли stack static→container или остался статикой.
1. **Web-app с нуля:** «CRM для записи клиентов: вход, список клиентов, добавление, заметки».
2. **Скрипт/инструмент (spa):** «калькулятор ипотеки с графиком».
3. **Телеграм-бот:** «бот для записи в барбершоп» (проверить — доступен ли вообще tgbot-стек: ранее orphaned).
4. **Свободный:** что-то из реальных запросов в логах прода (грепни недавние user-промпты).

### 2. ГЕНЕРАЦИЯ (через продукт)
- Драйв через браузер (авторизован) — gstack/playwright/Chrome MCP на `constructor.lead-generator.ru`: создать проект, отправить промпт(ы), дождаться сборки, открыть live-превью. ЛИБО через API минт-токеном (надёжнее headless): owner проекта-песочницы; токен — `docker exec omnia-prod-api /app/.venv/bin/python -c "from omnia_api.core.security import create_access_token; from uuid import UUID; print(create_access_token(UUID('<owner>')))"`, api на localhost:8200 (с VPS). Публичное превью: `<slug>-dev.preview.lead-generator.ru`.
- Снять **скриншоты** результата (desktop+mobile, light+dark если есть): сохранить в `_routine/runs/<ISO>/`.
- Скачать сгенерённый артефакт (HTML/.tsx/файлы) для анализа.

### 3. ОЦЕНКА (две оси)
- **Функциональность:** совпал ли результат с запросом? static vs реальный апп (есть `/app`-роуты, auth, БД, интерактив)? Для H1-репро — ЭСКАЛИРОВАЛ ли стек на follow-up? Кнопки/ссылки живые или тупики? Бот/скрипт реально делает заявленное?
- **Дизайн:** скриншот → оценка (rubric: иерархия, глубина, палитра-под-нишу, типографика, «не серый дефолт»). Можно прогнать через vision-судью продукта или оценить самому по скриншоту. Балл 0-10 + 2-3 конкретные проблемы.

### 4. EVIDENCE с сервера (корреляция «что увидел» ↔ «что решил сервер»)
- SSH прод, по project_id/slug грепнуть логи `omnia-prod-api` / `omnia-prod-worker`:
  - какой stack выбрал discovery, какой `gen_mode` (freeform/static/catalog), какая модель, был ли re-discovery на follow-up;
  - `[VSEGPT] stream usage … cache_hit=…` (кэш B), ошибки/traceback, пустые брифы, таймауты.
- Зафиксировать ИМЕННО серверное решение, которое привело к видимому провалу.

### 5. ДИАГНОЗ + СЛЕПЫЕ ЗОНЫ
- Назвать тупняк одним предложением + file:line, где решение принимается неверно.
- **Слепые зоны:** чего генератор НЕ проверяет/НЕ ищет (напр. не перечитывает интент на follow-up; не эскалирует стек; не зовёт судью; не видит, что страница — мёртвая статика с нерабочими кнопками).

### 6. УЛУЧШЕНИЕ (по доказательствам, не вслепую)
- Если фикс ЯСНЫЙ, безопасный, доказан логом+скриншотом → реализовать МАЛЫМ диффом + тест-репро → доставить (commit+push main+деплой нужных сервисов) → health-check. Цитировать строку лога/скриншот, что доказывает.
- Если фикс крупный/рискованный → НЕ шипить вслепую: записать PROPOSAL в LEDGER (что, почему, какой evidence, какой план).
- Один прогон — максимум один фикс (последовательность, не ломать прод).

### 7. ЗАПИСЬ
- Append строку в RUNS (ниже): время MSK, сценарий, stack/gen_mode/модель, design-балл, тупняк, действие (fix `<commit>` / proposal / none), PASS/FAIL.
- Сырьё (скриншоты, дампы, полный лог прогона) — в `_routine/runs/<ISO>/` (gitignored).
- Снять lock.

Корень (три поверхности, код-доказан): (1) `createRecord` (engine.ts:217-226) только zod-парсит body + `insert`, не грузит target-сущность для проверки существования referenced-строки; (2) `deleteRecord` (engine.ts:295-315) удаляет по id, возвращает `{deleted:true}`, не сканит детей, нет restrict/cascade/warn; (3) `expandRecords` (engine.ts:143-148) `_expanded[field]=map.get(refId) ?? null` — мёртвый родитель неотличим от «связь не задана» → silent «—». **Почему не фикшу вслепую:** референциальная целостность — это СЕМАНТИЧЕСКОЕ решение политики, меняющее write/delete-поведение КАЖДОГО генеримого аппа, и правильная политика реально неоднозначна:
1. **RESTRICT** — блокировать delete, пока существуют дети, → 409. Безопасно для данных, но ломает «просто удалить» UX и требует сканить ВСЕ сущности на reference-поля, целящие в эту (граф связей), на каждый delete.
2. **CASCADE** — удалять детей вместе. Деструктивно, легко снести больше, чем юзер ожидал.
3. **SOFT/WARN** — вернуть affected-children count и требовать confirm. Кросс-зонно (apps/web + CrudResource confirm-UX).
Все варианты мульти-поверхностные (engine + apps/web), требуют base-rebuild + regen-verify, и data-integrity-чувствительны (сейчас тихая порча данных). Плюс отдельный пробел: нет валидации ref на write (bogus-ref принимается). Max один фикс/прогон, не блайнд-шип.

**План безопасного фикса (для отдельного прогона/согласования):**
1. Выбрать политику delete (RESTRICT по умолчанию безопаснее всего для CRM; CASCADE опционально per-relation; WARN как UX-надстройка). Для RESTRICT — построить обратный индекс «кто ссылается на эту сущность» из всех `entities/*.json` reference-полей, на delete запросить детей → при наличии 409 с affected-count.
2. Валидация ref на write: createRecord/updateRecord — для reference-поля проверить, что target-строка существует и доступна (scope-aware), иначе 400.
3. `expandRecords` — отличать «ref задан, но мёртв» (флаг dangling) от «ref не задан», чтобы UI мог показать «удалён» вместо «—» (опционально).
4. Health-check БЕЗ генерации: пересобрать base, throwaway omnia-dev-* с Client+Note, DELETE Client с живой Note → restrict 409 (или cascade удаляет Note); write с bogus-ref → 400.

**Acceptance-lock (уже в репо):** `apps/orchestrator/tests/test_entity_referential_integrity.py` — `test_create_does_not_validate_reference_targets_today` + `test_delete_is_unconditional_with_no_child_scan_today` + `test_expand_silently_nulls_a_dead_reference_today` (3 green evidence) + `test_deleting_a_referenced_record_should_not_silently_orphan_children` (xfail strict=False — XPASS, когда движок станет референциально-осведомлённым на delete).

**Проблема (доказана live + кодом, BS-25):** entity-движок не имеет оптимистической блокировки. `updateRecord` (engine.ts:285-297) читает существующую строку, делает `merged = {...existing, ...parsed.data}` и пишет целиком; WHERE — только id+entity+createdBy, без version/updated_at-предусловия → не замечает, что строка изменилась с момента загрузки редактором. Форма усугубляет: edit-режим `<EntityForm>` сидит каждое поле из загруженной записи (entity-form.tsx:82-88) + `validate()` эмитит каждое непустое (entity-form.tsx:149-167) → каждое сохранение = полный устаревший снапшот, а не патч → устаревшее поле реально шлётся и затирает чужую правку. **Почему не фикшу вслепую:** оптимистическая блокировка — это смена write-контракта КАЖДОГО генеримого аппа, мульти-поверхностно:
1. **schema/init-db:** version-колонка в `records` (или использовать `updated_at` как токен).
2. **engine:** `updateRecord` принимает `If-Match`/`expectedVersion`, добавляет предусловие в WHERE (или сравнивает version перед записью) → 409 на mismatch вместо тихого затирания.
3. **form/SDK:** грузит и шлёт текущую version при сохранении.
4. **apps/web/CrudResource:** conflict-resolution UX («запись изменилась с момента открытия — перезагрузить / перезаписать»), иначе 409 = просто новая мёртвая стена.
5. **Регрессия:** наивное предусловие сломает легитимные последовательные правки одного редактора (version должна корректно инкрементиться/возвращаться). base-rebuild + regen-verify по нишам. Data-integrity-чувствительно. Один фикс/прогон.

**План безопасного фикса (для отдельного прогона/согласования):**
1. Выбрать токен: отдельная version-колонка (чище) ИЛИ `updated_at` как If-Match (без миграции, но грубее).
2. engine: предусловие + 409 `EngineError(409, "stale")`; вернуть актуальную строку в теле конфликта, чтобы UI показал «их» версию.

## PROPOSAL P-KPICOUNT — server count/aggregate-примитив для дашборд-KPI (2026-06-17, run #23)

**Проблема (доказана live + кодом + реальным сгенерённым аппом, BS-26):** дашборд entity-аппа считает каждую headline-цифру КЛИЕНТСКИ как `.length`/`.reduce()` над массивами `useEntity`/`entities[X].list()`, которые грузятся БЕЗ `limit` → движок отдаёт DEFAULT_LIMIT=50. Live-репро (throwaway omnia-dev-dogfood-kpicount-probe из DEPLOYED base, 60 Task): GET без params = 50 строк, тело = голый `{"data":[...]}` БЕЗ `total`-поля и БЕЗ `X-Total-Count`-хедера; `?limit=200`=60; `/count`|`/stats`|`/aggregate`→500 (нет count-роута). Реальный сгенерённый CRM (demo-crm-dlia-prosmotra `dashboard/page.tsx`): `entities.Deal.list()`/`Client.list()`/`Task.list()` все без limit + `value={activeDeals.length}`, `<CountUp value={newClientsThisMonth.length}/>`, `<CountUp value={overdueTasks.length}/>`, `avgCheck=deals.reduce(...)/deals.length`. Кит САМ учит этому (dashboard-hero.tsx:71-72, count-up.tsx:13) → структурно. Итог: «Активных сделок: 50» навсегда, средний чек/конверсия считаются над новейшими 50 = скошены. Это ПЕРВЫЙ экран при логине, цифра выглядит авторитетной — хуже тихого обрыва BS-18 и обрезки экспорта BS-24.

**Почему не фикшу вслепую:**
1. **Нет примитива.** Движок не умеет считать кардинальность без выборки всех строк (кап MAX_LIMIT=200). Нужен НОВЫЙ server count (`SELECT count(*)` с теми же scope/filter-условиями, что `listRecords`) — это новая поверхность в engine + API-роут или `{rows,total}`-envelope + SDK `count()` + `useEntity` `total`. Меняет list-data-flow контракт всех аппов.
2. **Count ОДИН не чинит avg/sum/ratio.** avgCheck/conversion/выручка-фаннел — это server SUM/AVG/GROUP BY, а не count. Даже с верным total средний-над-загруженными-50 останется скошенным. Правильный фикс = маленькая aggregate-поверхность (count + sum/avg по числовому полю + group-by-enum для фаннела), а не band-aid `limit:200` (двигает обрыв + грузит 200×ref на каждый KPI).
3. **Writer-слой вероятностен + кит учит `.length`.** Надо ОДНОВРЕМЕННО дать примитив И переписать kit-гайд (dashboard-hero/count-up JSDoc + SYSTEM_PROMPT) на «KPI = server count/aggregate, не `rows.length`», иначе writer продолжит `.length`. TEMPLATE+engine → base-rebuild + regen-verify. Тесно связан с P-PAGINATE/P-SEARCH/P-SORT/P-EXPORT (вся windowing-ремедиация) — логично делать вместе.

**План безопасного фикса (для отдельного прогона/согласования):**
1. engine: `countRecords(def,user,params)` (count(*) с scope+filter) и/или `aggregateRecords` (sum/avg/group-by по whitelisted-полю); вернуть `{rows,total}`-envelope из list ИЛИ отдельный `/api/entities/[entity]/count` роут.
2. SDK: `entities[X].count(params)` + `aggregate`; `useEntity` отдаёт `total` (и опц. агрегаты) для KPI-чтения.
3. Kit + промпт: dashboard-hero/count-up JSDoc + SYSTEM_PROMPT → MANDATORY «headline-KPI = server count/aggregate, НЕ `loadedRows.length`»; обновить few-shot дашборда.
4. Health-check на песочнице dogfood-*: 60+ записей → KPI показывает 60 (не 50); средний/сумма считаются по всему набору.

**Acceptance-lock (уже в репо):** `apps/orchestrator/tests/test_entity_kpi_count_windowing.py` — `test_engine_list_returns_a_bare_capped_page_with_no_total_today` + `test_sdk_list_returns_bare_array_with_no_count_method_today` + `test_use_entity_exposes_only_a_capped_rows_array_today` + `test_kit_jsdoc_teaches_length_over_loaded_arrays_for_kpis_today` + `test_no_count_or_stats_route_exists_today` (5 green evidence) + `test_runtime_should_expose_a_server_count_primitive_for_kpis` (xfail strict=False — XPASS, когда появится server count/aggregate-примитив).

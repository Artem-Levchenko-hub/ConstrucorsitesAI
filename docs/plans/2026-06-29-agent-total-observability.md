# Агент: тотальная наблюдаемость («разбирать любую проблему до атома»)

> Дата: 2026-06-29. Цель (owner): агент видит ВСЁ, логирует всё, разбирает любую
> проблему до атома; гейт проделывает путь юзера (логин→действие→проверка). Режим
> «мощно, но безопасно»: блок-гейт пока только realtime, остальное — наблюдаемость.
> Катим инкрементами, каждый — отдельный безопасный деплой.

## Корневая причина, которую закрываем
Баг «сообщение не отправляется» агент не чинил 3 раза, потому что **все его «глаза» были зелёные, а провал невидим**: typecheck зелёный (text/body — оба валидный TS), `runtime_check` бьёт GET и видит 200, `see` видит пустой чат. Единственный сигнал — `400 «empty message»` на POST при отправке, которую ни один инструмент не делает и не читает. А функц-гейт (`functional_gate.py:234`) публикует серверным контрактом `{data:{text}}` — ПРАВИЛЬНЫМ полем — поэтому проходит, пока сгенерённый UI шлёт `{body}` и падает. **Гейт тестирует подложку, не реальный UI.**

## Инкремент 1 — СДЕЛАНО (этот коммит)
- **Громкие диагностики на realtime publish-route** (`templates/nextjs-realtime/.../api/realtime/[channel]/route.ts`): каждый отказ (`403 forbidden`, `400 empty-message`, `429`) пишет `console.warn` с причиной и **списком ключей, что реально прислал клиент** → `read_logs` (он у агента уже есть) показывает точную причину мгновенно.
- **Инструкция в `LOOP_PROTOCOL`** (`agent_builder.py`): для ИНТЕРАКТИВНОЙ фичи (отправка/сохранение/логин) чистая сборка и 200-страница НЕ доказывают работу — после правки `read_logs` и ищи 4xx/5xx с причиной; чини пока запрос самого действия не 2xx.
- Прощающий контракт (`text|body|строка`) уже выкачен ранее (`65ad293`) — этот класс багов закрыт на подложке.

## Инкремент 2 — СТАТУС
- **2.1 `probe` — СДЕЛАНО.** Аутентифицированный E2E-инструмент (`services/agent_probe.py`), действие `probe` в allowlist+verify-actions+executor+LOOP_PROTOCOL. Агент делает РЕАЛЬНЫЙ запрос как залогиненный юзер и видит точный статус+тело.
- **2.3 `see` → консоль/сеть — СДЕЛАНО.** `preview.capture_diagnostics` + `agent_vision.see_page`: упавшие (>=400) запросы и JS-ошибки в наблюдении; упавший запрос → `see` ok=False (агент не закончит над сломанным fetch).
- **2.2 `sql` — ОТЛОЖЕНО ОСОЗНАННО.** `probe GET /api/.../messages` читает БД через API — то же «легло ли в БД». Чистый SQL избыточен и рискованнее (прямой доступ к БД проектов); вернуться, только если понадобится сырой доступ мимо API.
- **2.4 хард-блок гейт — ОТЛОЖЕНО ОСОЗНАННО.** E2E flaky → ложный блок ХОРОШЕЙ realtime-сборки = та же фрустрация, что чинили. probe+see+логи дают агенту глаза ПОЧИНИТЬ во время сборки (лучше, чем блок постфактум). Гейт остаётся heal-петлёй (уже есть). Включать хард-блок только после калибровки на флаки.

### Детали (если возвращаться)

### 2.1 `probe` — аутентифицированный E2E-инструмент (главный)
Новое действие агента `probe` → api-хелпер, переиспользующий **проверенный** `functional_gate` (Playwright + `_login` через NextAuth csrf+callback, `_api` = fetch внутри страницы с куками). Агент задаёт `{method, path, body, as?}` → получает `{status, json, text}` РЕАЛЬНОГО запроса как залогиненный юзер. Тогда `probe POST /api/realtime/<ch> {data:{body:"hi"}}` → видит `400 empty message` на 1-м шаге.
- Файлы: новый `services/agent_probe.py` (вынести `_login`/`_api` из `functional_gate.py` в общий хелпер `services/preview_session.py`), новая ветка в `make_container_executor._execute` (`agent_builder.py:1106`), `probe` в `_KNOWN_ACTIONS` + `LOOP_PROTOCOL`, контракт-тест.
- Риск: Playwright на каждый probe тяжёлый; ограничить кол-во вызовов (как verify-actions) + таймаут.

### 2.2 `sql` — read-only SELECT по схеме проекта
Новое действие `sql {query}` → orchestrator endpoint, выполняет SELECT под **read-only ролью проекта** (она уже есть, `postgres_admin`), возвращает строки. Проверка «легло ли в БД». Только SELECT (reject INSERT/UPDATE/DELETE/DDL парсером).
- Файлы: orchestrator `/agent/sql` (`routers/runtime.py`), `orchestrator_client.agent_sql`, ветка в executor, allowlist+protocol+test.
- Риск: инъекция/запись — строгий read-only (отдельная роль + парс-гард).

### 2.3 `see` → консоль + сеть браузера
`agent_vision.see_page` + `preview.capture_live_url`: навесить Playwright-листенеры `page.on("console")` + `page.on("response")`, вернуть в observation ошибки консоли и 4xx/5xx-ответы. Браузерный глаз (JS-ошибки, упавшие fetch).
- Оговорка: `see` неаутентифицирован → не доходит до /chat; для интерактива нужен авторизованный preview (2.1 решает).
- Файлы: `agent_vision.py:64`, `workers/preview.capture_live_url`.

### 2.4 Блок-гейт по РЕАЛЬНОМУ UI (realtime, потом шире)
Расширить `functional_gate` сценарием по сгенерённому UI: открыть /chat залогиненным, найти поле ввода + кнопку, отправить, проверить, что текст появился у ВТОРОГО клиента. Это ловит client/server-mismatch ({body} vs {text}). Сделать финальный FAIL **блокирующим** для realtime + фидить причину в heal-цикл.
- Оговорка: разметка генерится → хрупко; искать поле по `role=textbox`/`form`, на провал поиска — degrade к API-проверке (не вечный блок).
- Файлы: `functional_gate.py` (+UI-сценарий), wiring `messages.py:2971-3052` (block, не warning), флаг.

## Принципы (чтобы «безопасно»)
- Каждый инструмент **fail-soft**: ошибка → безобидная observation, не падение цикла.
- Блок-гейты bounded + degrade (не вечная блокировка хорошей сборки).
- Verify-инструменты (`probe`/`sql`/`see`) exempt от global-repeat guard (как build/runtime_check).
- Включать по флагу с мгновенным откатом; realtime первым, расширять после обкатки.

# Omnia.AI — тотальный анализ продукта для презентации

> Дата: 2026-06-28. Подготовлено по реальному коду (`apps/api`, `apps/orchestrator`, `apps/llm-gateway`, `docs/plans`) + веб-ресёрч по конкурентам.
> Формат: тезисы под слайды. Каждый факт о коде подтверждён ссылкой `file:line`.

---

## ГЛАВНЫЙ ТЕЗИС ПРЕЗЕНТАЦИИ (повторить 3 раза)

> **Потолок продукта — не модель, а harness (обвязка).**
> Почти всё для «без ограничений» **уже написано в коде**. Но 80% мощных механизмов **выключены за флагами** (default `False`).
> Сдвинуться = не «построить заново», а **«включить + докрутить 3 узких места»**.

Это подтверждено живым опытом: мессенджер на дешёвом DeepSeek собрался надёжно, когда починили harness (контракт-карта примитивов, ship-green-on-abort, sanitizer layout), а не модель.

## Слайд 1 — Сдвиг парадигмы: ДО → ПОСЛЕ

### ДО (одношотовая генерация — «text-генератор»)
- Промпт → модель **за 1 оборот** пишет весь код → regex парсит → файлы коммитятся.
- **Обратной связи ноль.** Модель не видит: ошибки компиляции (`TS2305`), runtime-краши (route 500), сломанную навигацию.
- Надёжность «угадал/не угадал»: **~40–60%** на сложных задачах.
- Для веб-приложения был только `nextjs_entities` — managed-абстракция, **не экспортируемый** настоящий бэкенд.

### ПОСЛЕ (агентная инженерия — «реальный инженер») — ✅ УЖЕ РАБОТАЕТ
- **Цикл план → действие → наблюдение → починка**, движок `agent_builder.py` (796 строк, рабочий).
- **11 инструментов:** `list_dir, read_file, grep, write_file, edit_file, build, bash, read_logs, runtime_check, see, done` (`agent_builder.py:31-44`).
- Текстовый протокол (не native function-calling) → **работает с любым шлюзом** (DeepSeek/Kimi/Claude).
- Модель видит **реальный мир**: typecheck-ошибки, HTTP-статус маршрута, скриншот рендера → читает ошибку → правит файл → пересобирает.
- Цикл-брейкеры (no-write streak, repeat-guard, stall→escalate cheap→strong), green-abort rescue (если на abort билд чистый — шипим).
- Расчётная надёжность с циклом: **~85–95%** при достаточном бюджете шагов.

### Ключевое отличие (одна картинка для слайда)
```
БЕЗ инструментов:  [модель] ───────────────► файлы (могут быть сломаны)
С ИНСТРУМЕНТАМИ:   [модель] ─► действие ─► РЕАЛЬНЫЙ результат
                       ▲                          │
                       └──── читает ошибку, правит, повторяет ◄──┘
```

> **Подвох для честности:** агентный цикл написан и работает, но **выключен по умолчанию** — `use_agentic_builder=False` (`config.py:805`). Это «тестовый режим / канарейка», не общий прод.

## Слайд 4 — ПРОБЛЕМА №1: правки трудно применяются

### Корень (тезисы)
1. **Механизм правки — byte-exact SEARCH/REPLACE**, не diff и не AST (`file_extractor.py:1010`). Модель должна **дословно** скопировать кусок исходника, и движок заменяет его.
2. **Дешёвая модель (`deepseek-chat`) портит дословную копию** — теряет символ в длинной строке классов (`min-h-[95vh]` → `min-h-[95h]`), путает отступ/кавычку → блок **не находится** → правка молча падает (`messages.py:3720-3735`).
3. **JSX в контейнерных апп особенно хрупкий** — деструктуризация, типы, callbacks: модель пишет «похожий, но не идентичный» код.
4. **Модель копирует из памяти, не из показанного файла** → SEARCH не совпадает с текущим содержимым.
5. **Якорь неуникален** (встречается 2+ раза) → matcher отказывается (безопасно, но теряет правку).
6. Иногда модель **забывает формат** — отдаёт голый HTML / `<file>` / болтовню вместо `<edit>`.

### Что уже сделано (митигации в коде, work)
- Эскалация `deepseek-chat` → `deepseek-v4-pro-thinking` при промахе (`messages.py:3689-3717`).
- 5 уровней fallback: surgical retry → zone-rewrite → full-page rewrite → **container rewrite** (переписать файл целиком, `messages.py:3972-4090`).
- Indent-tolerant матчинг + partial-apply (один промах не роняет остальные).
- `clean_chat_content` — честный чат: убирает фейковый чип «Правка», если правка не применилась.

### Как преодолеть (индустрия 2025-2026)
1. **Hashline-разметка** (самый дешёвый шаг, чистый harness-win): каждой строке — контент-хеш, модель ссылается на тег, **не копирует текст**. Замер: Grok Code Fast **6.7% → 68.3% (×10)**, без смены модели, ~$300. → *прямое подтверждение «проблема в harness».*
2. **Sketch → Apply** (стандарт Cursor/Morph): сильная модель пишет «что менять» лениво (`// ... existing code ...`), дешёвая **apply-модель** детерминированно вмёрдживает. Точность ~98%.
3. **«Прощающий» матчинг** (Aider, 5–9 стратегий) + правка **крупными блоками** (целая функция) + **валидация билдом ДО записи** на диск.

## Слайд 5 — ПРОБЛЕМА №2: слабый дизайн/UX + «не сразу адаптивно»

### Корень (тезисы)
1. **Кто верстает:** `art_director` = Kimi K2.6 (дизайн-мозг), `freeform_writer` = **DeepSeek v4-pro** (`config.py:1087-1101`). DeepSeek — **дешёвый кодер, не художник**: он исполняет бриф, не творит.
2. **Нет живого дизайн-судьи ДО генерации.** Дизайн начинается с умозрительного брифа, не с перцепционной оценки.
3. **Mobile-first — это совет, а не требование.** Правило адаптива стоит на ~130-й строке промпта из 2000+ (`prompt_builder.py:130-138`) — модель уже выбрала пресет и оптимизировала desktop, прежде чем дочитала.
4. **Адаптив проверяется ПОСЛЕ генерации, не ДО.** Гейт рендерит 375/768/1440 и блокирует горизонтальный скролл (`acceptance.py:119-124`) — но это пост-факт «заплатка», а не mobile-first билд.
5. **Нет mobile-UX-метрик** в гейте: тач-таргеты ≥44px, line-height, минимальная ширина колонок — не проверяются.
6. **Модель игнорирует kit-классы** (`.reveal`, `.bg-mesh`, `.display-hero`) и пишет свой CSS — нет штрафа за переписывание кита.
7. **Vision-судья (оценка красоты по скриншоту) ВЫКЛЮЧЕН** — `use_vision_audit=False` (`config.py:246`); даже включённый — advisory + видит серые плейсхолдеры вместо фото.

### Как преодолеть (тезисы)
1. **Mobile-first сделать ТРЕБОВАНИЕМ и поднять в топ промпта** (позиция 10-15): «сначала 360px, потом `.md:`/`.lg:`». Адаптивность лидеров (v0) — от **ограничения стека** (Tailwind+shadcn), не от «ума» модели.
2. **Мобильный acceptance-гейт:** тач-таргеты ≥44px, line-height ≥1.5, ширина колонок ≥280px, `@container`-queries для дашбордов.
3. **Включить vision-audit по умолчанию** + ждать полной загрузки картинок перед скриншотом + поднять бюджет дизайн-судьи до 3-4 итераций.
4. **Усилить art_director** (Opus-класс) — один сильный бриф дешевле двух переделок.
5. **Штраф за игнор kit-классов** в self-check.

## Слайд 6 — Скрытая ПРОБЛЕМА №3: непредсказуемость (корень «то работает, то нет»)

> Это и есть техническая причина, почему «иногда отличный апп, иногда мусор».

- **Контейнерные апп (`fullstack`, `nextjs_entities`, `spa`, `tgbot`, `api`) ПОЛНОСТЬЮ пропускают acceptance-гейт** — подтверждено: `messages.py:4760` (`not in (...)`).
- Значит на настоящих веб-приложениях **не проверяется**: мёртвые ссылки, один `<h1>`, горизонтальный скролл, красота, runtime-5xx — **первая генерация шипится непроверенной**.
- Vision-судья на них тоже не запускается (нет статичного `index.html`).
- Единственная страховка — error-cards (компиляция/билд) в фоне, **advisory, не блокирует**.
- **Agent_builder** (единственный, кто видит реальные runtime-ошибки) — **выключен**.

**Итог:** статичные лендинги проходят 4-уровневый контроль качества, а **самые ценные апп (с бэкендом) — нет**. Отсюда ощущение «нестабильности».

## Слайд 8 — ПОЧЕМУ МЫ НЕ МОЖЕМ СДВИНУТЬСЯ (резюме, 1 слайд)

Три причины, и все три — **не «не построено», а «не включено / не докручено»:**

1. **Всё мощное — за флагами OFF.** Агентный цикл, vision-судья, контейнерный гейт, gVisor — написаны, но выключены из страха регрессий и стоимости. → Прод работает в «слабом» режиме одношота.
2. **Дешёвая модель — общий потолок.** Один и тот же DeepSeek — потолок и для **правок** (ломает byte-exact), и для **дизайна** (кодер, не художник). Лечится harness'ом (hashline, apply-модель, mobile-first enforce), а не «подождём модель получше».
3. **Самые ценные апп — вне контроля качества** (Bug #1). Непроверенная первая генерация = подрыв доверия = нельзя масштабировать.

## Слайд 9 — Дорожная карта преодоления (тезисно, по приоритетам)

### 🔴 Включить почти бесплатно (дни)
- **Hashline для правок** — ×10 надёжности правок, чистый harness-win.
- **Mobile-first сделать требованием** + поднять в топ промпта.
- **Контейнерный acceptance-гейт ON** (`USE_CONTAINER_ACCEPTANCE_GATE`) → закрыть Bug #1.
- **Vision-audit ON** для контейнерных апп.
- **`PHOTO_SOURCE=openverse`**, **gVisor/cap-drop ON** (Phase 1 уже в коде).

### 🟠 Докрутить (2–4 недели)
- **Agent_builder ON через канарейку** (`use_agentic_builder`) для container-билдов → реальная петля «до зелёного».
- **Sketch→Apply** (apply-модель) для правок.
- **Мобильный UX-гейт** (44px / line-height / ширина колонок).
- **ЮKassa** (реальные платежи).

### 🟢 Новые возможности
- **Android `.apk`** — реально и недорого по образцу живого `.exe`-пайплайна:
  - дефолт **PWA → TWA через Bubblewrap** (headless, без Android-разработчика);
  - build-контейнер с Android SDK (JDK 17 + cmdline-tools + platform-35 + Gradle) на runtime-сети, как Wine-контейнер;
  - env-keystore, `assetlinks.json` на деплое, артефакт `.apk` рядом с `.exe`;
  - флаг `USE_APK_BUILD` + doctor (`apksigner verify`). Второй трек — Capacitor (камера/push).
- **Экспорт проекта** (`tar.gz` + `docker compose up`) — Phase 5.
- **Авто-провижин настоящей БД без ключей + экспорт схемы/RLS** (фича Lovable Cloud) — как анти-lock-in продающий аргумент.

## Слайд 10 — Главный вывод для аудитории

- Omnia **уже умеет** строить настоящие full-stack веб-приложения с Postgres, real-time мессенджеры, Telegram-боты, REST API, Python-скрипты и **настоящие Windows `.exe`** — со своей инфрой, превью и деплоем.
- Переход к **агентной инженерии** (инструменты + петля проверки) — это сдвиг от «генератора текста, который надеется» к «инженеру, который делает, проверяет, чинит». Движок **готов**.
- Две боли (правки, дизайн/адаптив) + скрытая третья (непредсказуемость) — **это harness-проблемы**, решаются включением и докруткой готовых механизмов, а не сменой модели.
- Единственный реальный пробел из «без ограничений» — **Android `.apk`**, и путь к нему понятный и дешёвый.

> **Слоган слайда:** «Мы не строим с нуля — мы включаем то, что уже построено. Потолок — harness, а не модель».
